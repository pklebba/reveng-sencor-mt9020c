# Firmware — ESP32 IR bridge

A PlatformIO project that **transmits and receives** the Sencor MT9020C IR
protocol. It builds the verified frame on the MCU ([`src/main.cpp`](src/main.cpp)
— a C++ port of [`../tools/encode.py`](../tools/encode.py)), sends it via an IR
LED, and exposes a small HTTP API + web UI over WiFi. An optional IR receiver
lets the device decode the real remote (or its own transmission) and print the
bytes — handy for verification.

The IR carrier is generated by the ESP32 **RMT** peripheral (hardware). A
software bit-banged carrier (e.g. IRremoteESP8266) comes out at the wrong
frequency on the ESP32-C6 and is shredded by WiFi interrupts; RMT is exact and
interrupt-immune. No external libraries are used.

```
firmware/
├── platformio.ini        # boards (esp32c6 default, esp32dev fallback)
└── src/
    ├── main.cpp          # frame build + RMT TX + RMT RX/decode + HTTP server
    └── secrets.h.example # copy to secrets.h, set your WiFi (gitignored)
```

## Hardware

For the full IR-hub build (ESP32-S3 + MOSFET-driven LEDs + TSOP, with a bill of
materials and full wiring), see [`HARDWARE.md`](HARDWARE.md). Quick version below.

### TX — IR LED (required)
Drive the LED through a transistor; a GPIO pin can't source the current an IR
LED needs for usable range:

```
  GPIO4 ──[330 Ω]──► base ┐
                          │  NPN (2N2222 / BC337)
  5V  ──[~22 Ω]──►|──────► collector
       (TSAL6400)         │
                 emitter ─┴── GND
```

- **IR LED:** TSAL6400 (or any ~940 nm IR LED). A visible LED will **not** work.
- **Series R:** ~22 Ω from 5V (~33 Ω scaled for 3V3) for ~100 mA+. Driving from
  the board's 5V pin gives more range than 3V3.
- **Quick bench test:** the LED may sit directly on `GPIO4` via ~100 Ω a few cm
  from the AC's receiver — proof of concept only, too weak for real range.

### RX — TSOP receiver (optional, for debugging)
A 38 kHz IR receiver module on `GPIO5` lets the firmware decode incoming frames:

```
  TSOP  VCC → 3V3        (add 100 nF VCC↔GND close to the TSOP — it's noisy
        GND → GND         without decoupling and floods the receiver)
        OUT → GPIO5
```

Point the real remote (or the board's own IR LED) at the TSOP; decoded bytes
appear on serial and at `http://<ip>/rx`.

## Configure

1. Copy `src/secrets.h.example` → `src/secrets.h` and set `WIFI_SSID` /
   `WIFI_PASS`. (`secrets.h` is gitignored.)
2. Optional, in `src/main.cpp`: `IR_LED_PIN` (default 4), `IR_RX_PIN`
   (default 5), `carrierKhz` (default 38 — also tunable live via `?khz=`).

## Build & flash (PlatformIO / CLion)

Install **PlatformIO Core** (`pip install platformio`) and the **PlatformIO for
CLion** plugin, then open this `firmware/` folder.

```bash
cd firmware
pio run -t upload     # build + flash (default env: esp32s3)
pio device monitor    # serial @ 115200
```

The default board is **esp32-s3-devkitc-1** (the target IR-hub board — USB-C,
native USB, 16 MB flash). It uses the [pioarduino](https://github.com/pioarduino/platform-espressif32)
platform fork, which ships the Arduino core for every current chip. Two more envs
are kept for the boards used during bring-up — build either with `-e <env>`:

- **`esp32c6`** — ESP32-C6-DevKitC-1 (USB-Serial-JTAG, no brownout); the board the
  protocol was validated on.
- **`esp32dev`** — bare ESP-WROOM-32 + CP2102 (needs the manual bootloader dance).

### Status LED

Dev boards with an onboard addressable RGB LED (S3 `GPIO48`, C6 `GPIO8`) use it as
a status indicator: **green** = WiFi connected, **red** = WiFi down, **blue blink**
= frame transmitted. It compiles out on the bare WROOM (no such LED).

### Troubleshooting

- `ModuleNotFoundError: No module named 'intelhex'` while building
  `bootloader.bin` — install it into PlatformIO's Python:
  `~/.platformio/penv/bin/pip install intelhex`
- RX prints `L1 H1 L1 ...` noise — the TSOP needs a 100 nF VCC decoupling cap
  (a hardware glitch filter and software debounce mitigate it, but the cap is
  the real fix).
- Bare WROOM + CP2102: enter the bootloader manually (hold `IO0`→GND, pulse
  `EN`→GND, release, then upload). Add yourself to the `dialout` group.

## Control

Open `http://<esp-ip>/` for a click-to-send web form, or use the HTTP API:

```bash
curl "http://<esp-ip>/ac?power=on&mode=cool&temp=22&fan=auto"
curl "http://<esp-ip>/ac?mode=heat&temp=18&fan=high&swing=1&khz=38"
curl "http://<esp-ip>/off"
curl "http://<esp-ip>/rx"      # last frame decoded by the receiver
```

| Param | Values                            | Default |
| ----- | --------------------------------- | ------- |
| power | `on` / `off`                      | on      |
| mode  | `cool` `heat` `dry` `fan` `hold`  | cool    |
| temp  | °C (Heat 13–26, others 16–32)     | 23      |
| fan   | `auto` `sleep` `low` `med` `high` | auto    |
| swing | `0` / `1`                         | 0       |
| unit  | `c` / `f` (display only)          | c       |
| khz   | carrier in kHz                    | 38      |

Every send echoes the 14 transmitted bytes (serial and HTTP response), e.g.
`sent  frame=23 CB 26 01 00 24 03 09 00 00 00 00 00 45  carrier=38kHz`.

## Notes

- Frame format, LSB-first bytes and the checksum are documented in
  [`../docs/protocol.md`](../docs/protocol.md) and verified against the real
  remote.
- The carrier wasn't recoverable from the demodulated capture, so 38 kHz is an
  assumption — if a unit doesn't respond, sweep `?khz=36..40`. The frame timing
  and checksum are confirmed, so the carrier is the only unknown.
